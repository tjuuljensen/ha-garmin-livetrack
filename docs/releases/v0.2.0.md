# Garmin LiveTrack v0.2.0

## Summary
`v0.2.0` is the first release intended for stable tag-based HACS distribution.

This release moves the integration onto a stronger runtime model with:
- staged Garmin fetches
- a dedicated incremental trackpoint endpoint
- adaptive polling based on Garmin publish frequency
- per-session transport backoff
- data-driven activity normalization and icon mapping
- a cleaner Home Assistant options flow
- a clearer separation between integration responsibilities and automation responsibilities

## Highlights
### Incremental Garmin trackpoint fetching
The integration now prefers Garmin's dedicated incremental endpoint for trackpoints:
- `/api/sessions/{sessionId}/track-points/common`

The fetch strategy is layered:
1. fetch the public LiveTrack page
2. establish cookies and CSRF context
3. fetch session metadata
4. fetch incremental trackpoints
5. fall back to broader payload and hydration parsing if needed

This improves efficiency while keeping the existing resilience to Garmin response-shape changes.

### Adaptive polling and profile-based tuning
Global update behavior is now preset-driven:
- `Extended`
- `Conservative`
- `Balanced`
- `Adaptive`
- `Advanced`

`Adaptive` uses Garmin `postTrackPointFrequency` when available so the integration does not attempt trackpoint fetches before Garmin is likely to have published a new point.

`Advanced` opens a dedicated advanced path for low-level tuning and troubleshooting.

### Per-session transport backoff
The integration now applies bounded per-session backoff for Garmin/API failures:
- `429` backs off aggressively
- `5xx` and retryable request failures back off with a shorter bounded cooldown
- successful fetches clear backoff state

This reduces avoidable Garmin pressure during transient failures without bypassing lifecycle handling.

### Activity normalization and icon mapping
Activity handling is now data-driven.

The integration preserves both:
- `activity_type_raw`: Garmin's original value
- `activity_type`: normalized canonical value

Examples:
- `Trail Running` -> `running`
- `Mountain Biking` -> `cycling`
- `kayaking` -> `kayak`

Unknown Garmin activities remain visible and continue to work with a generic fallback icon.

### New automation event for incoming points
The integration now emits:
- `garmin_livetrack_point_received`

This event includes normalized point data such as:
- `activity_type`
- `activity_type_raw`
- `activity_icon`
- `speed_kmh`
- `pace_min_km`
- `distance_km`
- `duration_hms`

Raw Garmin tokens are not included.

### Notifications removed from the integration
The integration no longer sends notifications directly.

Instead, it exposes:
- entities
- device trackers
- services
- Home Assistant events

Notification delivery is now expected to live in Home Assistant automations, scripts, blueprints, or YAML packages.

### Improved diagnostics and repair signaling
The integration now exposes more useful diagnostics and repair signals, including:
- effective HTTP User-Agent
- adaptive scheduling state
- per-session backoff state
- shape-change suspicion and repair issue signaling

## User-visible changes
### Options flow
The options flow has been reorganized:
- normal use is preset-driven
- advanced tuning is explicit
- configured users now support suggestions plus free-text pre-registration
- user policy editing remains separate from advanced global tuning
- per-user removal is available directly in the Options UI

### Per-user status behavior
Per-user status sensors retain ended-session state and summary values during the configured retention window, including across Home Assistant restarts.

### User management
Strict-user handling now honors manually configured users from the configured-users list, not only users already present in stored policy state.

The integration also includes a direct per-user remove action in the Options UI, in addition to the existing service-level admin operations.

### Tracker display names
Per-user device trackers now use the Garmin user label directly as the entity display name, which improves map presentation and avoids generic `GLT` initials.

### Aggregate entity model
Global aggregate entities are attached to one integration device.
Per-user entities are attached to user devices.

## Compatibility notes
- Old notification-related integration settings are ignored and migrated away safely.
- Old stored notification policy fields are removed during migration.
- The integration continues to support IMAP event ingestion and manual URL submission.
- A service/IMAP startup bug in the add-url path caused by a missing initial-trackpoint-wait import has been fixed.

## Upgrade notes
After upgrading:
1. restart Home Assistant
2. reopen the integration options flow if you use custom tuning
3. verify any automations that consume Garmin LiveTrack events
4. if needed, run `garmin_livetrack.cleanup_legacy_entities` to remove stale entity-registry entries

## HACS release model
This release is intended to be published as a tagged GitHub release:
- tag: `v0.2.0`
- GitHub release title: `v0.2.0`

HACS should follow the tagged release rather than arbitrary branch state.
